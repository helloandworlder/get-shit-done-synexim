# GSD Synexim 项目协作约定

本项目默认采用 GSD Synexim 作为项目执行框架。

## 1. 语言与输出

- 所有面向用户的说明、规划文档、阶段总结、状态报告默认使用中文。
- 若项目代码、协议字段或第三方 API 必须使用英文，保留英文标识，但解释时必须使用中文。
- 完成文档或实现后，都要输出一个类图形化进度卡，至少说明：本次写了什么、涉及哪些模块、为什么这样做、下一步做什么、当前里程碑进度。

## 2. 规模自适应

- `micro`：微型任务或一次性小需求，优先使用 `/gsd:quick`，或只做一份轻量计划后立刻编码。
- `standard`：常规项目，使用`Phase -> Plan -> Task`。
- `large`：大型项目或多模块系统，使用 `Milestone -> Phase -> Plan -> Task`。
- 进行 Phase 划分时，必须遵守“人类驱动渐进式需求对齐”：每个 Phase 都必须交付一个或多个完整可用的基础功能，而不是只做技术分层切片。
- 禁止把阶段切成“先数据库、再 ORM、再 API、再前端”这种横向分层；正确方式是每个阶段都覆盖前端、后端、数据库、缓存和联调，至少让一个模块真正可用。
- 若需求是“实现/修复/开发”，最多只允许一次轻量规划，且必须进入代码，不要反复写文档。

## 2.1 模型策略

- 规划、路线图、需求收敛、Phase/Plan 设计默认使用 `OpenAI GPT-5.4`。
- 规划优先使用 `GPT-5.4 xhigh / Extreme High`；若上下文或成本不适合，则退到 `GPT-5.4 high`。
- 代码进入实现、联调、修复、验证时，统一使用 `GPT-5.4 high`。
- `GPT-5.4` 在项目初始化阶段不得承担任何前端设计工作；初始化阶段只负责需求澄清、原型检查、覆盖核对和实现准备。

## 3. 主动触发 GSD

出现以下情况时，Agent 应主动优先考虑 GSD：

- 用户要求梳理需求、路线图、阶段、计划、任务、文档结构
- 任务涵盖两个及以上模块
- 需要连续多步执行、验收或里程碑管理
- 需要记录 `PROJECT.md`、`REQUIREMENTS.md`、`ROADMAP.md`、`STATE.md`、`PHASE`、`PLAN`
- 用户希望看到进度、剩余任务、下一步建议

若只是极小修改，可直接执行，但仍应遵守中文输出和进度反馈要求。

## 4. 前端原型强制门禁

- 只要项目、里程碑或任务涉及前端页面、交互流程、仪表盘、表单、移动端页面或设计系统，必须先要求用户基于 `FEATURES.md`、`REQUIREMENTS.md`、`ARCHITECTURE.md` 等材料，在 Gemini AI Studio 生成详细原型。
- 前端进入实现前，`GPT-5.4` 必须确认已经存在“完整且与需求对齐的前端 MVP 原型”；若没有，则不得开始前端编码。
- 编码前必须检查原型是否100%覆盖一级功能，以及是否覆盖所有关键页面和空态、错态、成功态。
- 可主动启动 SubAgent 对比原型、`.planning` 文档和当前里程碑范围；若发现 “MVP 有但 .planning 没写” 或 “.planning 写了但 MVP 没有” 的差异，必须先向用户列出差异，再根据用户命令决定补全还是移除。
- 若原型缺漏页面、流程或状态：先指出缺口，再询问用户，并在本地启动前端 Demo 服务补齐 MVP 原型后再继续大规模编码。

## 5. 文档与执行平衡

- 文档必须服务于执行，不能为了“外观完整”而无限扩写。
- 每份 Phase / Plan / Summary 文档都必须明确包含“下一步立即执行项”。
- 若当前上下文足以安全实施，优先写代码、验证、修复，而不是继续抽象讨论。
- 每个 Phase 完成后必须停下，启动项目，告诉用户访问地址、登录方式、测试账号/种子数据（如有）、建议测试路径，并请求用户反馈后再继续后续 Phase。

## 6. 推荐的 GSD 入口

- 新仓库或首次接入：`/gsd:init`
- 新项目整理：`/gsd:new-project`
- 微型任务：`/gsd:quick`
- 既有仓库补丁：`/gsd:patch-agents`
- 查看状态：`/gsd:progress`

## 7. 进度卡格式

默认使用如下风格：

```text
┌─ 本次更新 ───────────────────────────────┐
│ 文档: ...                               │
│ 模块: ...                               │
│ 做了什么: ...                           │
│ 为什么: ...                             │
│ 下一步: ...                             │
│ 进度: ...                               │
└────────────────────────────────────────┘
```
