# GSD Synexim 项目协作约定

本项目默认采用 GSD Synexim 作为项目执行框架。

## 1. 语言与输出

- 所有面向用户的说明、规划文档、阶段总结、状态报告默认使用中文。
- 若项目代码、协议字段或第三方 API 必须使用英文，保留英文标识，但解释必须使用中文。
- 每次完成文档或实现后，都要输出一个类图形化进度卡，至少说明：本次写了什么、涉及哪些模块、为什么这样做、下一步做什么、当前里程碑进度。

## 2. 规模自适应

- `micro`：微型任务或一次性小需求，优先使用 `/gsd:quick`，或只做一份轻量计划后立刻编码。
- `standard`：常规项目，使用 `Phase -> Plan -> Task`。
- `large`：大型项目或多模块系统，使用 `Milestone -> Phase -> Plan -> Task`。
- 若需求是“实现 / 修复 / 开发”，最多只允许一次轻量规划，随后必须进入代码，不要反复写文档。

## 3. 主动触发 GSD

出现以下情况时，Agent 应主动优先考虑 GSD：

- 用户要求梳理需求、路线图、阶段、计划、任务、文档结构
- 任务跨越两个及以上模块
- 需要连续多步执行、验收或里程碑管理
- 需要记录 `PROJECT.md`、`REQUIREMENTS.md`、`ROADMAP.md`、`STATE.md`、`PHASE`、`PLAN`
- 用户希望看到进度、剩余任务、下一步建议

若只是极小修改，可直接执行，但仍应遵守中文输出和进度反馈要求。

## 4. 前端原型强制门禁

- 只要项目、里程碑或任务涉及前端页面、交互流程、仪表盘、表单、移动端页面或设计系统，必须先要求用户基于 `FEATURES.md`、`REQUIREMENTS.md`、`ARCHITECTURE.md` 等材料，在 Gemini AI Studio 生成详细原型。
- 编码前必须检查原型是否 100% 覆盖一级功能，以及是否覆盖所有关键页面和空态、错态、成功态。
- 若原型缺漏页面、流程或状态：先指出缺口，再询问用户，并在本地启动前端 Demo 服务补齐 MVP 原型后再继续大规模编码。

## 5. 文档与执行平衡

- 文档必须服务于执行，不能为了“看起来完整”而无限扩写。
- 每份 Phase / Plan / Summary 文档都必须包含明确的“下一步立即执行项”。
- 若当前上下文足以安全实施，优先写代码、验证、修复，而不是继续抽象讨论。

## 6. 推荐的 GSD 入口

- 新仓库或首次接入：`/gsd:init`
- 新项目梳理：`/gsd:new-project`
- 微型任务：`/gsd:quick`
- 既有仓库补丁：`/gsd:patch-agents`
- 查看状态：`/gsd:progress`

## 7. 进度卡格式

默认使用如下风格：

```text
┌─ 本次更新 ───────────────────────────────┐
│ 文档: ...                               │
│ 模块: ...                               │
│ 做了什么: ...                           │
│ 为什么: ...                             │
│ 下一步: ...                             │
│ 进度: ...                               │
└────────────────────────────────────────┘
```
